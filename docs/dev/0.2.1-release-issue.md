# CUDA Wheel Size Regression: v0.2.1 (99 MB) -> v0.2.2 (449 MB)

## Summary

The `inferna-cuda12` wheel's `libggml-cuda.so` grew from 99 MB (v0.2.1) to 449 MB uncompressed / 201 MB compressed (v0.2.2). The root cause is that adding `GGML_NATIVE=OFF` for portability caused llama.cpp's CMake to fall back to its full default CUDA architecture list, without `manage.py` passing an explicit `CMAKE_CUDA_ARCHITECTURES` to constrain it.

## Evidence

### v0.2.1 wheel contents (working, 99 MB)

```text
libggml-cuda.so sections:
  .nv_fatbin    86 MB
  .text         10 MB
  Total         99 MB

SASS: sm_52 only (130 ELF entries — 1 arch x 130 kernels)
PTX:  sm_52 only (130 PTX entries — 1 arch x 130 kernels)
```

### v0.2.2 wheel contents (bloated, 449 MB)

```text
libggml-cuda.so sections:
  .nv_fatbin   433 MB
  .text         10 MB
  Total        449 MB

SASS: sm_86, sm_89 (260 ELF entries — 2 archs x 130 kernels)
PTX:  sm_50, sm_61, sm_70, sm_75, sm_80 (650 PTX entries — 5 archs x 130 kernels)
```

### Key observation

The `.text` section (host code) is 10 MB in both versions. The instruction set flags (`GGML_AVX`, `GGML_FMA`, etc.) are **not** the bloat source. The entire regression is in `.nv_fatbin` — the CUDA fat binary section containing compiled GPU code for multiple architectures.

## Root Cause

Both v0.2.1 and v0.2.2 used llama.cpp's **default** CUDA architectures because `scripts/manage.py` never passed `CMAKE_CUDA_ARCHITECTURES` through to CMake (despite the CI workflow setting it in the environment). The HIP equivalent (`CMAKE_HIP_ARCHITECTURES`) was handled correctly — CUDA was simply missed.

The difference is what those defaults resolve to:

| Version | `GGML_NATIVE` | CMake behavior | Architectures compiled |
|---------|---------------|----------------|----------------------|
| v0.2.1  | ON (default)  | Detects CI runner GPU | sm_52 only (1 arch) |
| v0.2.2  | OFF           | Cannot detect target, uses full default list | sm_50, sm_52, sm_61, sm_70, sm_75, sm_80, sm_86, sm_89 (8 archs) |

When `GGML_NATIVE=ON`, llama.cpp's CMake detects the build machine's GPU (sm_52 on the GitHub Actions runner) and compiles for just that architecture with a PTX fallback for forward compatibility. When `GGML_NATIVE=OFF`, there's no GPU to detect, so CMake falls back to compiling for every default architecture — producing 5x more PTX and 2x more SASS.

## Commit timeline

| Commit | Change | Effect on wheel size |
|--------|--------|---------------------|
| `0.2.1` tag | No `GGML_NATIVE` set, no `CMAKE_CUDA_ARCHITECTURES` passthrough | 99 MB (native detection picks sm_52 only) |
| `05b232a` "set GGML_NATIVE=OFF" | Added `GGML_NATIVE=OFF` for portability | 449 MB (all default archs compiled) |
| `8be93c9` "fix cude wheel build-size" | Added `GGML_NATIVE_DEFAULT=OFF` | No effect (not a real CMake variable) |
| `f1adaf9` "v0.2.2hotfix" | Explicitly disabled `GGML_AVX/AVX2/FMA/F16C/BMI2` | No meaningful effect (`.text` was already 10 MB) |
| `3557d29` "another attempt to fix" | Added `CMAKE_CUDA_ARCHITECTURES` passthrough in `manage.py` | Correct fix (not yet verified in CI) |

## Fix

The fix in commit `3557d29` adds the `CMAKE_CUDA_ARCHITECTURES` environment variable passthrough to all three `get_backend_cmake_options()` methods in `scripts/manage.py` (for llama.cpp, whisper.cpp, and stable-diffusion.cpp), matching the existing pattern for `CMAKE_HIP_ARCHITECTURES`.

With the CI's `CMAKE_CUDA_ARCHITECTURES="70-real;75-real;80-real;86-real;89-real;90"` actually reaching CMake:

- `-real` suffix = SASS only, no PTX (for sm_70, 75, 80, 86, 89)

- `90` without `-real` = PTX only for sm_90 (forward compatibility for future GPUs)

- Expected result: ~6 SASS + 1 PTX per kernel instead of 2 SASS + 5 PTX per kernel

Expected wheel size should be comparable to or smaller than v0.2.1's 99 MB uncompressed / ~45 MB compressed.

## Verification

v0.2.1 was tested successfully on an RTX 4060 (sm_89, CUDA 12.0):

```text
$ python -c "... LlamaModel + inference test ..."
Response: 2 + 2 = 4
v0.2.1 CUDA: PASS
```

The v0.2.1 wheel works despite only containing sm_52 SASS because the PTX for sm_52 is JIT-compiled by the CUDA driver to sm_89 at runtime. This incurs a one-time startup cost but is functionally correct.

## Notes on `GGML_AVX`/`GGML_AVX2`/etc. flags

The explicit `GGML_AVX=OFF`, `GGML_AVX2=OFF`, `GGML_FMA=OFF`, `GGML_F16C=OFF`, `GGML_BMI2=OFF` flags added in the fix commits are still correct for **portability** — they prevent SIGILL crashes when the wheel runs on machines lacking those instruction sets. However, they are not the cause of or solution to the size regression. They should be kept for correctness, not for size.
