## [Unreleased]

### Added

- `GenerationConfig` (`src/inferna/api.py`) gained thirteen sampler knobs that were previously bound at the C++ layer but unreachable from the high-level API: `frequency_penalty` (default 0.0, disabled), `presence_penalty` (default 0.0, disabled), `penalty_last_n` (default 64; -1 = full context, 0 = disabled), `mirostat` (default 0; 0 = off, 1 = v1, 2 = v2), `mirostat_tau` (default 5.0), `mirostat_eta` (default 0.1), `typical_p` (default 1.0, disabled — locally-typical sampling), `typical_min_keep` (default 1), `xtc_probability` (default 0.0, disabled — "Exclude Top Choices"), `xtc_threshold` (default 0.1), `dynatemp_range` (default 0.0, disabled — dynamic temperature; > 0 swaps `add_temp` for `add_temp_ext`), `dynatemp_exponent` (default 1.0), and `logit_bias` (default None — OpenAI-shape `Dict[int, float]` of token-id-to-bias). The previously-existing `repeat_penalty` is now bundled into the same `add_penalties(last_n, repeat, freq, present)` call rather than being the only knob with a hardcoded window of 64 and zeroed siblings, so e.g. setting `frequency_penalty=0.5` while leaving `repeat_penalty` at its default no longer silently drops the penalty stage. Validation: `mirostat` ∈ {0,1,2}, `mirostat_tau`/`mirostat_eta` > 0, `penalty_last_n` >= -1. Sampler-chain behaviour: `logit_bias` (when non-empty) lands first, right after `add_grammar`, so per-token nudges sit on top of the grammar mask but before truncation/penalty stages reshape the distribution; penalties (when any of repeat/freq/presence is non-default) are added before the sampling tail; when `mirostat != 0` and `temperature > 0`, the tail becomes `temp` -> `add_mirostat` (v1, takes `n_vocab` and a fixed `m=100` words-considered) or `add_mirostat_v2`, replacing top-k/top-p/min-p/dist; greedy mode (`temperature == 0`) bypasses mirostat since the two are incompatible. In probabilistic mode (`temperature > 0`, `mirostat == 0`), the new knobs slot into the existing `min_p -> top_k -> top_p` ordering at upstream-compatible positions: `typical_p` between `top_k` and `top_p` (skipped when `typical_p >= 1.0`), `xtc` after `top_p` (skipped when `xtc_probability == 0.0`), and `add_temp_ext` replaces `add_temp` when `dynatemp_range > 0` (otherwise plain `add_temp`). Existing `min_p`/`top_k`/`top_p` order is preserved for behavioural compat — upstream uses `top_k -> typical_p -> top_p -> min_p`, but reordering would silently change sampling output for current users. Same chain logic mirrored across the three sampler-build sites: `LLM._create_sampler` (`src/inferna/api.py`), `BatchGenerator.generate_batch` (`src/inferna/batching.py`), and `Chat.__init__` (`src/inferna/llama/chat.py`) — the last of which previously hardcoded `add_penalties(64, repeat_penalty, 0.0, 0.0)` and now honours the new kwargs. New defaults exported from `inferna.defaults`: `DEFAULT_MIROSTAT`, `DEFAULT_MIROSTAT_TAU`, `DEFAULT_MIROSTAT_ETA`, `DEFAULT_MIROSTAT_M`, `DEFAULT_TYPICAL_P`, `DEFAULT_TYPICAL_MIN_KEEP`, `DEFAULT_XTC_PROBABILITY`, `DEFAULT_XTC_THRESHOLD`, `DEFAULT_DYNATEMP_RANGE`, `DEFAULT_DYNATEMP_EXPONENT` (the existing `DEFAULT_PENALTY_LAST_N` / `DEFAULT_PENALTY_FREQ` / `DEFAULT_PENALTY_PRESENT` are now actually wired through). Validation: `typical_p` and `xtc_probability`/`xtc_threshold` ∈ [0, 1]; `dynatemp_range >= 0`, `dynatemp_exponent > 0`; `logit_bias` keys must be `int` (rejecting `bool`), values must be numeric. End-to-end smoke-tested on `Llama-3.2-1B-Instruct-Q8_0`: typical_p, xtc, dynatemp, logit_bias (objective check — banned `" the"` token at greedy temp; output substituted `" a"` as expected), mirostat v1/v2, and combined-knob configs all produce coherent output. All 50 chat + batching tests still pass.
- Whisper streaming transcription (`inferna.whisper.streaming.WhisperStreamer` and the iterator helper `transcribe_stream`). whisper.cpp's encoder consumes a full mel spectrogram per call -- there is no native incremental decoder -- so the streamer implements the standard rolling-window re-pass pattern from upstream's `examples/stream`: callers `feed(samples)` chunks of mono float32 16 kHz audio (any chunk size); the streamer buffers up to `length_ms` of audio (default 10s) and runs `whisper_full` over the rolling window every `step_ms` of newly-accumulated audio (default 3s, matching the upstream default). Each pass emits `StreamSegment(text, t0, t1, is_final)` records in stream-relative time. `is_final=True` marks segments whose audio has rolled past the trailing edge (next pass's window won't see them, so their text is committed); `is_final=False` marks the in-flight tail that may be revised by subsequent passes. A monotonic watermark (`_final_until_sec`) suppresses re-emission of regions already committed -- the same audio region is never finalised twice. `flush()` runs a final pass that marks all remaining segments final and closes the streamer. `transcribe_stream(audio_iter, ...)` wraps the class for the common iterator-of-chunks shape (file readers, mic callback queues). Validation: `length_ms < step_ms` raises `ValueError`; non-float32 / non-1-D input is auto-coerced via `np.asarray`. Tests: 10 in `tests/test_whisper_streaming.py` covering arg validation, silence handling (no false-positive transcripts on pure-zero audio), JFK end-to-end (canonical phrase fragments present in the joined finals after chunked feed), iterator helper parity, monotonic non-overlapping finals, post-close `feed()` raises, and context-manager close. Wall-clock: 4.2s for the full file using the bundled `models/ggml-base.en.bin` and `tests/samples/jfk.wav`. Module-level `np` import (numpy is a project dep declared in `pyproject.toml`); `np = self._np` lazy-stash pattern dropped after a build/typecheck pass surfaced the unnecessary `Any`-leak.
- `LLM` now caches the seq-0 KV state across calls and reuses the prefix when a follow-up prompt overlaps the previous (prompt + generated) turn. Implementation: every successful call updates `self._kv_seq_tokens` (a list of token ids reflecting whatever currently lives in the live context's seq 0 — prompt followed by everything we sampled). On the next call, `_generate_stream` computes the longest token-id prefix shared with the new prompt, drops the divergent tail via `ctx.memory_seq_rm(0, overlap, -1)`, and prefills only the suffix. The last prompt token is always reserved for fresh decode so `sample(ctx, -1)` reads up-to-date logits, even when the new prompt is exactly the cached prompt. The shadow is cleared on `reset_context()`, on `close()`, on context recreation (size grew), on cancellation mid-prefill (KV may be torn), and inside `_apply_loras_to(ctx)` (a different adapter set produces a different attention model, so prior KV is invalid). Cache hits are correctness-preserving (replay of the same KV positions) so determinism under a fixed seed is identical to a full prefill — verified by a regression test in `tests/test_prompt_cache.py`. Measured speedup on a 200-token prefix with 10 generated tokens: **8.2×** vs. cold-start prefill on the default Llama-3.2-1B test model. Tests: 9 new in `tests/test_prompt_cache.py` covering shadow tracking, the perf fast path (must be ≥1.5× faster than full prefill), determinism under cache hit, partial-prefix hits, full-prefill on no-overlap, and shadow invalidation on `reset_context` / `close` / `clear_loras`. The `_ensure_context` reuse path no longer calls `kv_cache_clear()` automatically — it had to go to make the prompt-cache layer effective; the embedding context (separate `LlamaContext` instance, owned by `LLM._embed_ctx`) still clears its own KV per `embed()` call.
- `LLM(..., logprobs=True)` and `LLM(..., top_logprobs=K)` for token-level logprobs. When `logprobs=True`, every generated token gets a `TokenLogprob` record on `Response.logprobs` carrying the sampled token's text, id, and the log-softmax of the raw model logits at the sampling step (`log p(token | context)` under the *untransformed* model distribution — sampler-side temperature/top-k/top-p/grammar/penalty transforms reshape the sampling distribution but not the reported logprob, matching OpenAI's contract). When `top_logprobs > 0`, each entry also carries the top-K alternatives at that step, in descending logprob order, picked via `heapq.nlargest` so the cost is O(n log k) rather than the O(n log n) of a full vocab sort (matters on 128k+ vocabs). `top_logprobs > 0` implies `logprobs=True`. EOG tokens are not recorded (matches OpenAI: the stop token doesn't appear in the logprobs array). New dataclasses `TokenLogprob` / `TopLogprob` (frozen) added at module level. New `Response.logprobs: Optional[List[TokenLogprob]]` field. Mutual exclusion enforced: `logprobs=True` + `stream=True` raises `NotImplementedError` (per-chunk logprobs would land in the iterator; deferred until streaming-shaped demand emerges); `top_logprobs < 0` raises `ValueError`. Logprob calls bypass the response cache because cached `Response` objects don't carry logprobs that match the current `top_logprobs` setting. Implementation: `_build_token_logprob` computes a numerically-stable log-softmax (subtract max-logit before `exp`, sum exps, take log) over `ctx.get_logits_ith(-1)`, picks top-K via `heapq.nlargest`, and resolves token ids to text via the existing `_safe_piece` helper. Tests: 9 new in `tests/test_logprobs.py` covering arg validation (negative `top_logprobs`, mutex with `stream=True`, `top_logprobs > 0` implies `logprobs=True`), shape correctness (one entry per generated token, finite logprobs ≤ 0), `top_logprobs` ordering (descending) and dominance (top-1 raw logit ≥ sampled token's logprob to within numerical-stability ε), capping at vocab size when over-asked, and the cache-bypass contract.

### Changed

- Internal restructure: split the ~2070-LOC `LLM` mega class in `src/inferna/api.py` along three clean seams. Public API on `LLM` is unchanged — every previously-existing method keeps its signature and observable behaviour; the implementations are now thin delegations onto three new internal modules. (1) `src/inferna/_internal/response_cache.py` (`ResponseCache`, `ResponseCacheInfo`, `make_cache_key`): the LRU+TTL cache that previously lived inline as `_ResponseLRUCache` plus the cache-key derivation. `make_cache_key` takes the random-seed sentinel as a parameter rather than importing `LLAMA_DEFAULT_SEED` directly so the module stays import-cycle-free. `inferna.ResponseCacheInfo` is now re-exported from this module via the package `__init__`. (2) `src/inferna/_internal/chat_template.py` (`apply_template`, `get_template`, `format_messages_simple`, `_apply_jinja`): the two-tier (Jinja → C-API → simple) fallback ladder, previously duplicated across `LLM._apply_template`/`_apply_jinja_template`, the standalone `apply_chat_template`/`_apply_jinja_template_standalone`, and `inferna.llama.chat.Chat._apply_template`. All three call sites are now thin wrappers; the bare `except Exception: pass` arm in the standalone path was narrowed to `(TypeError, KeyError, AttributeError)` with `logger.warning`, matching the (previously-fixed) ladder used by the LLM instance method. The standalone `apply_chat_template` and `get_chat_template` free functions are also collapsed onto the helper. (3) `src/inferna/_internal/mcp_facade.py` (`MCPFacade`): the lazy `McpClient` lifecycle plus transport-type inference (`command` => stdio, `url` => http), fail-fast connect on `add_server`, idempotent disconnect on `remove_server`, and best-effort `close`. `LLM._mcp_client` is replaced by `LLM._mcp = MCPFacade()`; the public methods (`add_mcp_server`, `remove_mcp_server`, `list_mcp_tools`/`_resources`, `call_mcp_tool`, `read_mcp_resource`) are one-line delegates. The empty-list short-circuits in the read-only methods (no client constructed yet) move to the facade, matching prior behaviour exactly. Net: `api.py` shrank from ~3000 to ~2517 LOC; the `LLM` class shrank from ~2070 to ~1680 LOC. Test surface: `tests/test_api_mcp.py` was updated to construct `MCPFacade()` and reach into `llm._mcp._client` (rather than `llm._mcp_client`); `tests/test_jinja_chat.py` got a small `_apply_jinja_via_module(llm, ...)` helper that calls `inferna._internal.chat_template._apply_jinja(llm.model, ...)` in place of the now-removed `LLM._apply_jinja_template`. All 1519 tests still pass.

### Added

- `LLM(...)` now accepts OpenAI-compatible `tools=` and `tool_choice=` for function calling. Wire format mirrors OpenAI's chat-completion API exactly: `tools=[{"type": "function", "function": {"name": ..., "description": ..., "parameters": <JSON schema>}}, ...]`. `tool_choice` accepts `"required"` (must call a tool), `"auto"` (model picks tool vs. plain content; default), `"none"` (caller should omit `tools=`), or `{"type": "function", "function": {"name": "x"}}` (constrain to one specific tool). Implementation routes through the `response_format=` plumbing landed earlier this cycle: `compile_tools` builds a JSON schema (oneOf across tools, each branch fixing `name` to a `const` literal so `arguments` cannot be valid for the wrong tool) and reuses `CompiledResponseFormat` so there's no second sampler-builder code path. Auto mode wraps the choice in a `{tool_call|content}` envelope; the validator routes the parsed result onto `Response.tool_calls` (a new field, `Optional[List[ToolCall]]`) or onto `Response.text` when the model picked the content arm. Tool-call ids match OpenAI's `call_<24 hex>` shape; `arguments` is JSON-serialised string per the wire format. New module: `src/inferna/_internal/function_calling.py` (`compile_tools`, `CompiledToolResult`, `FunctionCall`, `ToolCall`). New top-level re-exports: `inferna.FunctionCall`, `inferna.ToolCall`. Mutual exclusion enforced: `tools=` + `response_format=` raises `ValueError`; `tools=` + `stream=True` raises `NotImplementedError` (streaming tool calls would need OpenAI-style incremental `arguments` deltas, deferred). Tests: 23 new in `tests/test_function_calling.py` covering grammar generation across all three tool_choice modes, validator routing for required/specific/auto, error paths (empty tools list, missing function name, duplicate names, unknown tool_choice, tool_choice='none', wrong tool_choice type, unknown function name in specific-choice), and live integration against the default test model (required/specific/auto + Response.tool_calls=None when `tools=` absent + mutex assertions).
- `LLM(...)` now accepts `response_format=` for structured outputs. Three input shapes are supported: `{"type": "json_object"}` (any well-formed JSON, constrained by the standard json.gbnf grammar inlined at `src/inferna/_internal/structured.py:_ANY_JSON_GBNF`), `{"type": "json_schema", "schema": <dict | BaseModel>}` (strict schema), and a pydantic `BaseModel` subclass directly (shorthand for the previous form). When supplied, generation is grammar-constrained via the existing `LlamaSampler.add_grammar` chain (grammar added FIRST so it filters the logit set before temperature/top-k/top-p run over the filtered set), and the resulting text is validated and attached to a new `Response.parsed` field — `dict` for plain-schema / `json_object` modes, a populated model instance for pydantic. Validation failures (model hits `max_tokens` mid-structure, etc.) leave `parsed=None` rather than raising, so callers can decide to retry. Compiled grammars are cached: dict schemas via `lru_cache(maxsize=128)` on a `json.dumps(..., sort_keys=True)` key (so two semantically equal dicts share the slot — verified by identity test in `tests/test_structured.py`); pydantic models in a small `id()`-keyed dict so `model_json_schema()` runs once per class. Plain calls (no `response_format`) leave `Response.parsed` at `None` and continue to use the response cache; structured calls bypass the cache because pydantic validators don't reliably round-trip through pickle. New module: `src/inferna/_internal/structured.py` (`compile_response_format`, `CompiledResponseFormat`). Tests: 18 new in `tests/test_structured.py` covering compile-time validation (None, json_object, json_schema dict/pydantic, malformed inputs), cache identity, pydantic validator success/failure, and live integration against the default test model.
- `LLM.load_lora(path, scale=1.0)`, `LLM.unload_lora(adapter)`, `LLM.clear_loras()`, `LLM.list_loras()` (`src/inferna/api.py`): high-level LoRA adapter management. `load_lora` constructs the adapter via `LlamaModel.lora_adapter_init(path)`, appends `(adapter, scale)` to `self._loras`, and pushes the new apply-list onto the live context (or defers until the first generation triggers `_ensure_context`, which now ends with `_apply_loras_to(ctx)` so adapters survive `reset_context()` and the auto-resize). `unload_lora` raises `ValueError` if the handle was never applied (rather than silently no-opping). The model owns each adapter's underlying memory; the LLM holds references via `self._loras` and drops them in `close()`. Tests: 4 in `tests/test_lora_embed.py::TestLoraStateManagement` plus a context-wiring test that exercises the empty-list re-apply path through a real generation.
- `LlamaContext.set_adapters_lora(adapters, scales)` (`src/inferna/llama/_llama_native.cpp`): native binding for `llama_set_adapters_lora`. Takes parallel lists of `LlamaAdapterLora` handles and `float` scales (validated equal-length, no-null-handle), throws `std::runtime_error` on a non-zero return. Passing empty lists clears the apply-set. The list-replace semantics (rather than per-adapter add/remove) match upstream's contract: each call replaces the full set previously attached to the context.
- `LLM.embed(text_or_texts, pooling="mean"|"cls"|"last", normalize=True)` (`src/inferna/api.py`): top-level embedding API. Lazily constructs a sibling embedding context (separate from the generation context, since `set_embeddings_mode` and the default KV state must be set at context creation), tokenises with `add_special=True`, decodes once per input with all-token output flags so per-token hidden states are available, pools in Python (consistent with the existing `inferna.rag.Embedder` strategy — llama.cpp's internal pooling has historically been unreliable for some architectures used in embedding mode), and L2-normalises by default. Single `str` returns a flat `list[float]`; a `list[str]` returns `list[list[float]]`. Empty list returns `[]`. Caps inputs at `model.n_ctx_train`. The embedding context is sized to the longest input on first use and reused across calls; `close()` drops it. Tests: 9 in `tests/test_lora_embed.py::TestEmbed` covering shape, normalisation magnitude, pooling-strategy invocation, empty-list path, cross-call context reuse, generation-not-disturbed-by-embedding, and recreate-after-close.
- `LlamaVocab.token_to_piece_bytes(token, lstrip=0, special=False)` (`src/inferna/llama/_llama_native.cpp`): raw-bytes counterpart to `token_to_piece`. Returns `nb::bytes` directly from the `llama_token_to_piece` buffer with no UTF-8 decoding. Streaming callers feed the result through `codecs.getincrementaldecoder("utf-8")` so codepoints split across token boundaries — common with byte-level BPE tokenizers — are joined cleanly. The existing `token_to_piece` (which decodes with `errors="replace"`) is retained for non-streaming callers and for backwards compatibility.

### Fixed

- `Makefile` now detects Python correctly on Windows. The previous `SYSTEM_PYTHON := $(shell command -v python3 ... || command -v python ...)` probe relied on the POSIX `command` builtin, which isn't available in the default Windows shell GNU Make spawns — so `SYSTEM_PYTHON` resolved to an empty string and every `manage.py`-driven target (`build`, `setup`, `info`, `clean`, the per-backend `build-*` / `wheel-*` matrix, `reset-webui`) crashed with "missing program" instead of building. The detection now branches on `$(OS)` (set to `Windows_NT` by Windows itself, regardless of the shell `make` runs under) and hardcodes `SYSTEM_PYTHON := python` on Windows — Microsoft Store and python.org installers both register `python` on `PATH` and there is no `python3` shim. The Unix branch is unchanged: `command -v python3` then `command -v python`, preserving the existing behaviour on macOS and Linux.
- `tests/test_rag_loaders.py::TestDirectoryLoaderSymlinks` (4 tests: `test_symlink_to_file`, `test_symlink_to_directory`, `test_broken_symlink_skipped`, `test_symlink_outside_directory`) no longer hard-fails on Windows machines without Developer Mode or admin privileges. `Path.symlink_to` on Windows calls `CreateSymbolicLinkW`, which requires `SeCreateSymbolicLinkPrivilege` — held by Administrators and by users with Developer Mode enabled, but not by ordinary user accounts — and otherwise raises `OSError: [WinError 1314] A required privilege is not held by the client`. The class now carries a `@pytest.mark.skipif(not _symlinks_supported(), reason=...)` decorator backed by a module-level runtime probe (`_symlinks_supported()` writes a temp file and tries `Path(tmp).symlink_to(target)` once at import time, returning `False` on `OSError` / `NotImplementedError`). The probe is preferred over a static `sys.platform == "win32"` check because (a) it correctly *runs* the tests on Windows machines that do have the privilege (Developer Mode users, CI runners running as admin), and (b) it correctly *skips* on the rare POSIX setup where symlink creation is blocked (some sandboxes, fakeroot environments). The four tests now report SKIPPED rather than FAILED on a stock Windows user account; macOS and Linux behaviour is unchanged.
- `LLM._generate_stream` (`src/inferna/api.py`) and `BatchGenerator.generate_batch` (`src/inferna/batching.py`) no longer corrupt multibyte UTF-8 codepoints that span token boundaries. The previous `vocab.token_to_piece(...)` path used `PyUnicode_DecodeUTF8(..., "replace")` inside the C++ binding, so a token containing a partial UTF-8 sequence (lone lead byte or continuation byte — routine output of byte-level BPE) was silently turned into `U+FFFD` replacement characters at the seam, even though the bytes from the next token would have completed the codepoint. The `except UnicodeDecodeError` arms were dead code under that decoder. Both loops now call the new `token_to_piece_bytes` and feed the bytes through a per-stream `codecs.getincrementaldecoder("utf-8")(errors="replace")` instance, which buffers an incomplete tail until a full codepoint lands and flushes via `decoder.decode(b"", final=True)` after the loop. In the single-stream path the pre-yield batch advancement is preserved when the decoder returns `""` so KV-cache positions stay aligned with token positions. Stop-sequence matching now operates on the true decoded text rather than `U+FFFD`-poisoned strings, fixing a related miss where a stop string straddling two tokens could fail to trigger.
- `LLM._apply_template` (`src/inferna/api.py:1334`) and `Chat._apply_template` (`src/inferna/llama/chat.py:131`) no longer mask real bugs in the vendored Jinja path. Both fallback ladders previously caught `_JinjaTemplateError` (intentional, e.g. Gemma's `raise_exception('System role not supported')`) and then a bare `except Exception: pass` — which silently swallowed everything else, including `TemplateSyntaxError` on malformed templates, `NameError` from a typo in a future refactor, `ImportError` from a partial install — and degraded to the C-API path. The bare arm is now narrowed to `(TypeError, KeyError, AttributeError)` (the realistic render-context failures: missing variable, wrong type in messages, attribute lookup on `None`) and emits `logger.warning("Jinja chat-template rendering failed (%s); falling back to C-API path", type(exc).__name__)` so the degradation is observable rather than silent. Other exception classes propagate, surfacing genuine bugs at the source instead of routing them through a fallback that may silently produce different output.
- `BatchMemoryPool.get_batch` (`src/inferna/llama/_python_helpers.py:107`) no longer hands out half-reset batches when `batch.reset()` raises. The previous `try: batch.reset(); except Exception: pass` returned the same pooled batch even when reset failed, leaving the caller with a `LlamaBatch` in an undefined state — possibly with stale tokens, sequence ids, or logits flags from the prior request. The exception path now logs a `logger.warning` (`type(exc).__name__: str(exc)`) and falls through to constructing a fresh `_N.LlamaBatch(n_tokens=..., embd=..., n_seq_max=...)`, which is the same construction the empty-pool branch uses. The successful-reset path now returns inside the `try` so the fall-through is unambiguous.

## [0.1.2]

### Changed

- `inferna.llama.server.python.ChatRole` (`StrEnum`-equivalent — `class ChatRole(str, enum.Enum)` for the 3.10 floor): named replacement for the `"system"` / `"user"` / `"assistant"` / `"tool"` string literals previously sprinkled across `python.py:361,387-391` and `embedded.py:679,773,792-797`. Members are `str` subclasses so JSON serialization is unchanged (`{"role": "assistant"}` over the wire), equality with raw strings still holds (incoming request payloads — where clients send plain strings — interop without coercion), and they're usable as dict keys. `ChatMessage.role` keeps its `str` annotation so callers can pass either form. `TOOL` is included for forward compat with OpenAI's tool-call response messages even though `_messages_to_prompt` doesn't switch on it yet — having the constant defined now means tool-calling work just adds an `elif` rather than introducing a new string.
- `MongooseLogLevel` is now an `IntEnum` (`embedded.py`) instead of five module-level `_LL_*` constants. Members `NONE`/`ERROR`/`INFO`/`DEBUG`/`VERBOSE` mirror mongoose's `MG_LL_*`. `EmbeddedServer.set_mongoose_log_level(level)` accepts both the enum member and a bare int, so the `--mongoose-log-level {0..4}` CLI flag (which produces a plain int from argparse) works unchanged. The constants were `_`-prefixed because they started life as internal helpers; promoting them to a public enum gives callers IDE autocomplete and named comparisons (`level == MongooseLogLevel.ERROR`) instead of magic 0..4 ints.
- `EmbeddedServer` server logs are now high-signal by default. Mongoose's built-in tracer (every `accept_conn` / `read_conn` / `write_conn` / `mg_close_conn` event, plus a startup `MG_IO_SIZE: 16384, TLS: none` banner) was firing at level 3 (`MG_LL_DEBUG`) — a half-dozen lines per HTTP request, useless once the wire-format is known to work. `EmbeddedServer.__init__` now calls `Manager.set_log_level(MG_LL_ERROR)` *before* constructing the `Manager` (the banner is printed inside `mg_mgr_init` itself, so the lowering has to happen first), and the dispatcher emits one access-log line per request on the dedicated `inferna.llama.server.embedded.access` stdlib logger: `GET /props 200 285B 0.1ms`. Status and body size are tracked on `MongooseConnection` (new `__slots__` `status_code`, `body_size`); the streaming SSE path uses a local `_send` closure that increments `body_size` on each successful `send_chunk` so streamed responses report accurate totals (verified: `POST /v1/chat/completions 200 2402B 232.5ms` for an 8-token reply, byte-exact match to `wc -c` of the wire response). New `EmbeddedServer.set_mongoose_log_level(level)` helper and `--mongoose-log-level {0..4}` CLI flag let users restore the raw mongoose trace when debugging HTTP-level issues. Wire: new `inferna_mg_log_set` shim in `mongoose_wrapper.c` + `Manager.set_log_level` static method binding in `_mongoose.cpp`.
- `inferna info` now silences native-library log chatter on stdout/stderr. ### Fixed

- `EmbeddedServer` SSE chat completions are now actually streamed token-by-token to the wire, not just to the framing layer. Mongoose is a single-threaded poll-based event loop: the dispatch handler runs synchronously inside `mg_mgr_poll`, every `Manager.send_chunk` call only queues bytes into the connection's send buffer, and actual socket writes only happen on the *next* poll cycle — which can't run until the handler returns. Net result of the previous inline implementation: every SSE chunk piled into the send buffer for the full duration of generation, then flushed in one batch when the handler exited. Verified timing: all 36 chunks of an 8-token reply arrived within a 5ms window after a ~854ms wait — visually indistinguishable from a non-streaming response in the browser. Fix moves generation to a per-stream worker thread that pushes pre-formatted SSE frames into a `queue.SimpleQueue`; a new `_drain_streams()` pass runs from the main poll loop on every tick, pulls everything currently queued, calls `send_chunk`/`end_chunked` on the main thread (where mongoose expects all manager calls to come from), and detects client disconnect via `is_connection_alive` to flip the worker's `cancelled` flag at the next token boundary. New structures (`embedded.py`): `_StreamingState` dataclass (per-stream context — single-writer per side: worker owns `slot` + generation, main owns the queue's consumer end + `cancelled`-write); `EmbeddedServer._streams: dict[int, _StreamingState]` registry; `EmbeddedServer._stream_worker(state)` (no `Manager` calls — pure generation + queue push); `EmbeddedServer._drain_streams()`. `_stream_chat_completion` now opens chunked headers, sends the role-only opener synchronously (fast first-byte feedback), spawns a daemon thread, and returns immediately. The original ~80-line generation loop in the dispatcher is gone. Poll cadence in `wait_for_shutdown` dropped 100ms → 50ms so streamed chunks don't sit in the queue too long. The wire format is unchanged — same OpenAI `chat.completion.chunk` shape, same `[DONE]` terminator. Each stream now emits a separate access-log line on completion: `stream-done conn=<hex> model=<alias> bytes=<n> elapsed=<ms>` (or `stream-cancel` if the client dropped). Re-verified timing post-fix: 36 chunks of the same prompt now spread from 0.1ms (opener) through 875.9ms ([DONE]), with content tokens arriving at ~73ms cadence ≈ poll interval × token rate.
- `EmbeddedServer` chat completions no longer truncate mid-sentence on long answers. Both `_handle_chat_completions` paths defaulted to `max_tokens = request.max_tokens or 100`, which (a) capped every UI chat that didn't explicitly set `max_tokens` at 100 tokens — visibly mid-paragraph for any non-trivial response — and (b) silently broke the upstream webui's `max_tokens: -1` "unlimited" convention by feeding `-1` straight into `range(-1)`, producing zero tokens. New `EmbeddedServer._resolve_max_tokens(request)` helper maps `None`, `0`, or any negative value to `self._config.n_ctx` (the slot's existing `n_past >= context.n_ctx - 1` guard is the real upper bound; this just removes the artificial 100-token ceiling). A positive value is honored verbatim. Verified: a "long essay" prompt now produces 673 tokens with `finish_reason: stop` under both `max_tokens: -1` and `max_tokens: 0`; an explicit `max_tokens: 5` still caps at 5.
- `EmbeddedServer._dispatch` now logs and emits an access-log line even when the failure happens *before* the request handler runs (e.g. in `MongooseConnection.__init__`, or in any future bookkeeping added between request acceptance and `handle_http_request`). Previously the outer `try`/`except` only wrapped the handler call, so a constructor exception propagated to the C++ trampoline in `_mongoose.cpp:54-80` — which catches and 500s the client — but bypassed both the inferna error logger and the access log entirely. The access log gained a silent gap for any such request, and the actual exception was lost. The dispatcher now wraps the entire flow (construction + routing + handler + bookkeeping) in a single `try`/`except`, copies `status_code` / `body_size` into local variables inside the `try` (so the access-log call after the `try`/`except` works regardless of where the failure landed), uses `_logger.exception` for traceback capture, and inner-guards the fallback `send_reply` so a failure there doesn't suppress the access log on top. The C++ trampoline remains as a defense-in-depth catch.
- Wheels now ship the embedded server's web UI assets. The four gzipped files under `src/inferna/llama/server/assets/webui/` are build artifacts (regenerated by `scripts/manage.py:_copy_webui_assets`) and are gitignored — but scikit-build-core's git-aware package walker silently dropped them from the wheel as a result, so `pip install` would land a server that 404'd on every UI route. Added an explicit `file(GLOB) + install(FILES)` rule in `CMakeLists.txt:691-708` that installs `*.gz` from the assets directory into `inferna/llama/server/assets/webui/` in the wheel; emits `message(WARNING)` (not an error) when the directory is empty so a wheel can still be built before `make` populates it (useful for CI pipelines that build the sdist first). Verified: wheel now contains all four files (`index.html.gz` 3193 B, `bundle.css.gz` 288703 B, `bundle.js.gz` 1724611 B, `loading.html.gz` 181 B) at byte-exact sizes.
- `EmbeddedServer` slot reuse no longer fails with `decode: failed to initialize batch` after the first request. `ServerSlot.reset()` previously zeroed `context.n_tokens` but did not clear the underlying llama KV cache, so the second request — starting a new decode at position 0 while the cache still held positions `0..N-1` from the prior run — tripped llama's "sequence positions must remain consecutive" guard. `reset()` now calls `self.context.kv_cache_clear()` before zeroing `n_tokens`, which releases all cells across all sequences and is the correct slot-reuse semantics. Affects both streaming and non-streaming paths; verified across mixed orderings (non-stream → stream, stream → non-stream, repeated stream).
- `inferna_mg_http_reply_bytes` (`src/inferna/llama/server/mongoose_wrapper.c`) no longer emits an empty `Content-Length` header. Mongoose's `mg_printf` ships its own format-specifier subset that does **not** support `%zu`; passing `body_len` (`size_t`) through `%zu` produced `Content-Length: ` (empty value), which curl rejects with `Invalid Content-Length` and Chrome treats as a 0-byte body. urllib silently fell back to read-until-EOF, so the bug went undetected through Phases 1–3 testing. Cast to `unsigned long long` and use `%llu`, which mongoose does support.
- `src/inferna/llama/server/__main__.py` no longer fails to start the embedded server with `ImportError`. The CLI imported `from .mongoose_server import EmbeddedServer`, but no such module exists — the actual class lives in `.embedded`. The import path was stale from a pre-rename of the file. Fixed.
- `EmbeddedServer.start()` now releases all native state when bring-up fails. Previously, if `_mgr.listen()` returned False (or any later step raised), the server retained `_model` / `_slots` / `_embedder`, the manager's `_dispatch` closure, and the `signal.signal()` handlers — pinning the entire instance (and its `LlamaModel` + `LlamaContext` + `LlamaSampler` chain) to interpreter shutdown. That late teardown ran after Metal cleanup, tripping `GGML_ASSERT([rsets->data count] == 0)` and aborting the process. The cleanup is now a single `try`/`finally` keyed on a `success` flag: any non-success path drops the dispatcher handler, restores signal handlers, and clears refs to the loaded model/slots/embedder. Regression test: `tests/test_mserver_embedded.py::TestEmbeddedServerLifecycle::test_failed_start_releases_native_state`.
- SD preview callback no longer hands Python use-after-free wrappers. The C trampoline (`_c_preview_cb` in `_sd_native.cpp`) previously wrapped each `sd_image_t` with `owns=false`, retaining a raw pointer into the sd.cpp-owned buffer that gets reused or freed on the next sampling step. Any frame the Python callback stashed past the callback's return read poisoned/freed memory. Each frame is now deep-copied into a wrapper-owned buffer (`std::malloc`/`std::memcpy`, `owns=true`), matching the allocator convention used by `SDImage.from_numpy`. Regression test added in `tests/test_sd.py::TestPreviewCallback::test_preview_callback_frame_data_survives_callback_return`, backed by a new test-only `_test_invoke_preview_cb` binding that synthesizes a preview event and overwrites/frees the source buffer before returning.
- `Speculative.is_compat` no longer clobbers the target context's KV cache. The probe previously decoded two dummy tokens into seq 0 and then called `kv_cache_clear(True)`, silently destroying any prefill the caller had built. The probe now picks the first empty sequence id in `[0, n_seq_max)` and removes only that probe seq when finished, leaving caller data intact. When every sequence is occupied (typical for default `n_seq_max=1` with seq 0 prefilled), `is_compat` raises `RuntimeError` with an explanatory message rather than corrupting state. Two regression tests added in `tests/test_speculative.py`.

Previously scikit-build-core could pick up CMake 3.21 from the build env and pass it to a project that demands 3.26+, immediately failing configure with a confusing error.
- `pyproject.toml` `wheel.exclude` no longer contains `*.lib`. On Windows, delvewheel relocates DLLs into `inferna.libs/` and may copy companion `.lib` import libraries; stripping `*.lib` would silently delete them.
- `[tool.cibuildwheel.macos]` now declares its own `before-all` (running `manage.py build --all --deps-only`) and disables the global `before-build`, mirroring the existing Linux strategy. Previously macOS rebuilt all third-party C++ deps once per Python version (5x for cp310-cp314), and re-cloned the upstream repos each time.
- `Makefile` library-detection now uses a platform-conditional `SHLIB_EXT` (`dylib` on Darwin, `so` on Linux, `dll` elsewhere). `wheel-dynamic` and the `LIBLAMMA` definition both pick the right extension instead of hard-coding `libllama.dylib`.
- `CMakeLists.txt` skips backend `find_package(... REQUIRED)` calls (CUDAToolkit, Vulkan, hip, hipblas, rocblas, SYCL runtime, OpenCL) when `WITH_DYLIB=ON`. The pre-built ggml-* dylibs already have those backends linked; the `_llama_native` extension itself does not reference their symbols. Previously a developer building only the wrapper extension against a pre-built llama.cpp release still needed the full Vulkan SDK (etc.) installed.
- CI: `build-gpu-wheels.yml` and `build-gpu-wheels-abi3.yml` lifted their per-step `cibuildwheel==X.Y.Z` pin into a workflow-level `env.CIBW_VERSION` so the seven `pip install` invocations cannot drift apart. Previously `build-gpu-wheels-abi3.yml` had a `3.4.1` pin in the CUDA-Linux job while every other job in the same file used `3.4.0`.
- CI: `build-gpu-wheels.yml` cache keys now hash `pyproject.toml` in addition to `scripts/manage.py` and the workflow file. Brings the production workflow in line with the abi3 sibling and ensures `pyproject.toml` changes affecting build deps invalidate the cached `thirdparty/` tree.
- CI: `gh release create` invocations across all four release-uploading jobs (`build-cibw.yml`, `build-cibw-abi3.yml`, `build-gpu-wheels.yml`, `build-gpu-wheels-abi3.yml`) now distinguish "release already exists" from real failures. Previously the `2>/dev/null || true` pattern hid auth errors and rate-limit failures along with the expected "tag already exists" case, so a CI run could pass without uploading anything.

### Removed

- Vestigial `nb::object params` field on `LlamaModelW` and `nb::object params_obj` field on `LlamaContextW`. Both were declared with "keep alive" comments but never assigned.
- Unused `VERSION := 0.1.20` variable from `Makefile`. It was never substituted into any rule and never updated by `make bump`, so it could only mislead.
- `.github/workflows/build-new-wheels.yml`: half-finished staging clone with only one job (CUDA 13.1 Windows) and a `Jimver/cuda-toolkit@v0.2.35` pin that drifted from production's `v0.2.21`. Removed; future GPU variants should be staged via a feature branch on the production workflow rather than a parallel file.
- `scripts/funcs.sh` and `scripts/funcs_py.sh`: shell helper scripts referencing `./bin/llama-cli` (a path that has not existed since the rename to `thirdparty/llama.cpp/bin/...`) and `python -m inferna.chat` (wrong dotted path). Not referenced by any documented workflow. Deleted.
- `Makefile` `LLAMACPP_LIBS` and `MACOS_FRAMEWORKS` variables — defined inside the macOS-only block but never substituted into any rule.
- `manage.py` `STABLE_BUILD` flag: gated a branch with **identical** version pins in both arms, so the flag was a no-op. Replaced with direct `os.getenv` lookups (`LLAMACPP_VERSION`, `WHISPERCPP_VERSION`, `SDCPP_VERSION`, `SQLITEVECTOR_VERSION`) so per-component override actually works.
- `manage.py` `cython_cpp_files: list = []` no-op cleanup loop — leftover from the Cython era. Bindings have been hand-written nanobind sources for some time.
- `manage.py` `LlamaCppBuilder.extra_libs`: dropped `llama-common` from the list. `CMakeLists.txt` never linked the corresponding archive, and grep finds no `common_*` symbol referenced from any wrapper source.

### Fixed

- `LlamaContext` post-close crash surface: `n_ctx`, `n_ctx_seq`, `n_batch`, `n_ubatch`, `n_seq_max`, `pooling_type`, `encode`, `decode`, `set_n_threads`, `n_threads`, `n_threads_batch`, `set_embeddings_mode`, `set_causal_attn`, `install_cancel_callback`, `synchronize`, `get_state_size`, `kv_cache_clear`, all `memory_seq_*`, `get_logits`, `get_logits_ith`, `get_embeddings`, `get_embeddings_ith`, `get_perf_data`, `print_perf_data`, and `reset_perf_data` now raise `RuntimeError` if invoked after `close()` instead of dereferencing a null pointer.
- `WhisperContext` post-close crash surface: every native accessor (`n_vocab`, `n_text_ctx`, `n_audio_ctx`, all `model_*`, all `token_*`, `tokenize`, `encode`, `full`, all `full_get_*`, `print_timings`, `reset_timings`, etc.) now raises `RuntimeError` after `close()` instead of segfaulting on a null `whisper_context*`. Previously `close()` set the pointer to `nullptr` but no method gate enforced the lifecycle.
- `MtmdContext` lifecycle: `tokenize`, `encode_chunk`, `eval_chunks`, `get_output_embeddings`, and the `supports_*` / `audio_sample_rate` / `uses_*` properties now route through `ensure_valid()` instead of scattered `if (!s.ptr) throw` checks (or, for the prop accessors, silently returning `false`/`-1` on a closed context). Behavior is consistent with the other three context types.
- `manage.py` no longer silently uses a stale source checkout when the requested version changes. Each `build()` entry point (llama.cpp static + dynamic, whisper.cpp, stable-diffusion.cpp, sqlite-vector) calls a new `verify_checkout()` helper that compares `git rev-parse HEAD` against `git rev-parse <self.version>` and aborts with a clear "run `make reset`" message on mismatch. Previously the guard was `if not src_dir.exists(): self.setup()`, which meant a partial checkout from a prior run would silently take precedence over the new version pin.
- `_mg.send_reply()` (the unvalidated module-level reply primitive) accepted any `uintptr_t` and dereferenced it directly. A stale connection id (kept past request scope) or a fabricated value would trigger undefined behavior on freed memory. Replaced by `Manager.send_reply()` which walks `mgr.conns` and rejects pointers not in the live list (and pointers whose connections are mid-close).
- `EmbeddedServer.stop()` now restores the SIGINT/SIGTERM handlers it installed in `_setup_signal_handlers()`. Previously the bound `self._signal_handler` method registered with `signal.signal()` retained a strong reference to the EmbeddedServer, which retained the `_mg.Manager`, `LlamaModel`, every slot's `LlamaContext` + `LlamaSampler`, and any embedder model — all the way to interpreter shutdown. On macOS this tripped a `GGML_ASSERT([rsets->data count] == 0)` in `ggml_metal_free` because the Metal device deallocated while LlamaContexts were still alive. New regression test `TestEmbeddedServerNoSignalRetention.test_server_released_after_stop` pins the fix using `weakref.ref(server)` after `stop() + del + gc.collect()`.
- `make qa` now passes cleanly. Fixed a real runtime regression in `inferna/llama/llama_cpp.py` — a previous `ruff --fix` had stripped the `from ._python_helpers import (...)` and `from ._speculative import (...)` re-exports thinking they were unused, breaking `import inferna` at module load (`batching.py` imports `get_pooled_batch` / `return_batch_to_pool` through the `llama_cpp` facade). Restored the imports using the explicit `X as X` re-export form so static analyzers and linters recognize them as intentional. Plus 117 mypy strict-mode errors across 6 files: type annotations added to `_python_helpers.py` (helpers, n-gram cache, download API), `embedded.py` (signal-handler types, `__enter__`/`__exit__` signatures, dict generics), `stable_diffusion.py` (SDImage proxies, tuple-pair properties, callback wrappers); ruff E701/E702 (one-statement-per-line) fixes in `stable_diffusion.py` `save()` and `SDContextParams.__init__`; mypy override added for the nanobind-subclassing modules to relax the four strict-mode checks that fire purely because nanobind classes are seen as `Any`.
